01
Requirements Foundation
System-specific + common non-functional checks

Most bad designs happen because the candidate starts coding before fixing the scope. Requirements fall into two buckets: System-Specific and Common Miscellaneous.

1.1 System-Specific Requirements

These divide into two categories:

A. Physical / Structural
  • What objects exist in the real world?
  • What contains what?
  • One-to-one, one-to-many, many-to-many?
  • Can a child object exist independently?
  • Will it be queried independently?
B. Action-Based
  • Who performs this action?
  • What input is required?
  • What output is returned?
  • Which entities are read / modified?
  • What can go wrong?

Parking Lot — Actions by Actor

User
  • Enter parking lot
  • Get ticket
  • Park vehicle
  • Pay fee
  • Exit parking lot
Admin
  • Add floor
  • Add spot
  • Configure pricing
  • Mark spot unavailable
System
  • Assign nearest spot
  • Calculate fee
  • Update spot status
  • Generate ticket

1.2 Common Miscellaneous Requirements

1.2.1 Offline vs Online System

Offline System
  • Runs locally in one process
  • No API controller needed
  • No external database needed
  • main() acts as UI/client
  • Objects held directly in memory
eg.
TicTacToe CLI, Chess local game, Snake and Ladder
Online System
  • Controllers expose APIs
  • Services contain business logic
  • Multiple concurrent users
  • Thread-safety matters
  • IDs required for core entities
eg.
Parking Lot API, BookMyShow, Uber, Food Delivery
Client / UI
Controller / API Layer
Service / Facade Layer
Repository / Datastore Layer
Database / In-memory Store

1.2.2 Concurrency & Thread Safety

Matters When
  • Multiple users book same seat
  • Multiple drivers accept same ride
  • Multiple vehicles compete for same spot
  • Multiple payments update same booking
May Not Matter When
  • Clearly offline and single-player
  • Game loop processes one move at a time
  • Interviewer says: assume one request at a time

1.2.3 History, Undo, and Replay

Command Pattern
  • Each action is an object with execute/undo
  • Useful for undo/redo and audit logs
  • Memory efficient if undo logic is simple
Memento Pattern
  • Saves snapshots of object state
  • Useful for board games, editor state
  • Simpler if state is small

1.2.5 Exception Handling

IllegalStateException
  • Move after game is over
  • Exit without payment
  • Dispense before payment
IllegalArgumentException
  • Negative amount
  • Invalid board position
  • Invalid vehicle number
NotFoundException
  • User not found
  • Ticket not found
  • Booking not found
Custom Domain
  • ParkingFullException
  • SeatAlreadyBookedException
  • InsufficientBalanceException

1.2.6 Notifications / Observer Pattern

i
Use Observer when multiple subscribers need to react to the same event. Do not overuse for simple one-to-one responses — if a service method directly returns a result, no observer is needed.
If the event has multiple independent subscribers, I will use Observer/EventBus. If it is only a direct response to the caller, I will simply return the result.

Class Diagram / Layering
Layer responsibilities and production vs interview design

Layer Responsibilities

Controller
Receive requests, validate format, convert DTOs, call service, map to HTTP responses. No business logic.
Service / Facade
Expose use-case methods, orchestrate entities, apply business rules, handle consistency, call strategies/factories.
Core Entities
Hold domain state, encapsulate simple operations on own fields, maintain entity invariants.
Strategies
Encapsulate algorithms that can vary. Avoid large if-else blocks. Make behavior extensible.
Factories
Encapsulate complex object creation. Create subtypes based on a type flag.
Datastore / Repo
Store and retrieve entities. Hide in-memory vs DB details. Provide lookup methods.
Main / Driver
Instantiate everything, create sample objects, simulate API calls or user input.

Main / Driver Example

Java
public class Main {
    public static void main(String[] args) {
        DataStore dataStore = new InMemoryDataStore();
        SpotAssignmentStrategy assignmentStrategy = new NearestSpotStrategy();
        FeeCalculationStrategy feeStrategy = new HourlyFeeStrategy();

        ParkingLotService service = new ParkingLotService(
            dataStore, assignmentStrategy, feeStrategy
        );

        Vehicle vehicle = VehicleFactory.createVehicle(VehicleType.CAR, "V1", "AP01AB1234");
        service.parkVehicle(vehicle);
    }
}

Methods: Entity vs Service

Put in Entity
  • Acts on its own fields
  • Protects invariants
  • Not a multi-entity use case
Java
spot.isAvailable();
spot.occupy(vehicleId);
spot.release();
Put in Service
  • Coordinates multiple entities
  • Calls datastore / repository
  • Represents a use case
  • Depends on strategies
Java
service.parkVehicle(vehicle);
service.exitVehicle(ticketId);

Prefer Behavior Methods over Setters

Avoid This
Java
spot.setAvailable(false);
booking.setStatus(CONFIRMED);
Prefer This
Java
spot.occupy(vehicleId);
booking.confirm();

Object Reference vs ID Reference

Use Object Ref WhenUse ID When
Strong compositionStored independently
Child does not exist independentlyCan be queried independently
Not queried independentlyOne-to-many or many-to-many
Offline / in-memory systemDB-friendly design needed
For a pure in-memory version I can keep object references, but for DB-friendly design I will store IDs between aggregate roots.

Inheritance, Interfaces & Abstract Classes
When to use what
R
Simple Rule: Core entity with common state → abstract class. Pure behavior variation → interface. Fixed values with no behavior → enum.
Abstract Class
  • Subclasses share common state
  • Subclasses share common helper methods
  • Clear is-a relationship
  • Reduce duplicated fields/getters
Java
abstract class Vehicle {
    protected String id;
    protected VehicleType type;
    public abstract VehicleType getType();
}
class Car extends Vehicle { ... }
Interface
  • Represent behavior / capability
  • No shared state
  • Multiple inheritance of type
  • Strategy or policy
Java
interface FeeCalculationStrategy {
    Money calculateFee(Ticket ticket);
}
interface SpotAssignmentStrategy {
    ParkingSpot findSpot(...);
}

Common Mapping

EntityUseReason
Vehicle, Piece, ParkingSpotAbstract ClassCommon state (id, color, type)
FeeStrategy, PaymentStrategyInterfacePure behavior variation
VehicleType, BookingStatusEnumFixed values, no behavior

parkVehicle() Example

Java
public Ticket parkVehicle(String lotId, Vehicle vehicle) {
    ParkingLot lot = dataStore.getParkingLot(lotId)
        .orElseThrow(() -> new NotFoundException("Lot not found"));

    ParkingSpot spot = spotAssignmentStrategy.findSpot(lot, vehicle)
        .orElseThrow(() -> new ParkingFullException("No spot available"));

    spot.occupy(vehicle.getId());

    Ticket ticket = new Ticket(
        idGenerator.generate(), vehicle.getId(),
        spot.getId(), LocalDateTime.now()
    );

    dataStore.saveVehicle(vehicle);
    dataStore.saveSpot(spot);
    dataStore.saveTicket(ticket);

    return ticket;
}
The service will throw domain exceptions, and the controller/main layer will catch them and convert them into user-friendly responses.

Concurrency Checks Senior Signal
Race conditions, locks, atomic operations
!
Concurrency is one of the strongest signals of senior-level thinking in LLD. Race conditions usually happen in check-and-update operations.

Common Race Condition Scenarios

BookMyShow
  • Two users book the same seat simultaneously
Uber
  • Two drivers accept the same ride request
Parking Lot
  • Two vehicles assigned the same spot
Vending Machine
  • Two users buy the last unit of an item

Identification Template

text
API: acceptRide(driverId, rideId)
Modified entities:
- RideRequest: REQUESTED → ACCEPTED
- Driver: AVAILABLE → BUSY

Race risk:
- Two drivers may accept same ride
- Same driver may accept two rides

Protection:
- Lock ride request and driver
- Re-check status inside lock

Double-Checked Locking

Unsafe
Java
if (seat.isAvailable()) {
  synchronized (seat) {
    seat.book(userId); // stale!
  }
}
Safe
Java
if (seat.isAvailable()) {
  synchronized (seat) {
    if (!seat.isAvailable())
      throw new SeatAlreadyBookedException();
    seat.book(userId);
  }
}

ConcurrentHashMap vs Manual Locking

ApproachGood ForLimitation
ConcurrentHashMapIndependent key-based operations, high read/write concurrencyDoes not make multi-step business logic safe
Manual LockingMulti-step operations, updating multiple objects togetherLower throughput, must acquire locks in consistent order
synchronized methodSimple interview answer, prevents racesBlocks all unrelated operations
The risky part is the check-and-update section. Two requests may pass the check using stale state. I will protect this critical section by locking the affected entity or using an atomic update.

Mistakes to Avoid
Common anti-patterns in LLD interviews
Forgetting Related Entities
  • Always loop through all affected entities per write API
  • E.g., parking = spot + ticket + vehicle + availability count
Too Many Setters
  • Setters expose raw state, bypass invariants
  • Prefer booking.confirm() over booking.setStatus(CONFIRMED)
Logic in Controller
  • Controllers should only delegate to services
  • No pricing, locking, or booking logic in @PostMapping methods
Overusing Patterns
  • Factory for every class
  • Strategy for never-varying behavior
  • Singleton for every service
  • Observer for simple responses
Ignoring Partial Failure
  • Update multiple entities atomically
  • Validate first, then update in a small critical section
All Logic in Service
  • Service orchestrates; entities protect their own invariants
  • Anemic entities with only setters = bad design
